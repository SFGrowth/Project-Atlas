# Atlas Digital Twin Specification v1.0

**Sprint:** 080  
**Date:** 10 July 2026  
**Author:** Manus AI  
**Status:** Approved for Implementation

This document specifies the architecture of the Atlas Digital Twin. The Digital Twin acts as the canonical, real-time representation of the entire Atlas ecosystem. It abstracts the raw data generated by Pine Script and the Observatory into a unified, inspectable object model.

---

## 1. The Digital Twin Architecture

The Digital Twin is the central state management layer of the Atlas Nexus Operating System (ANOS). No dashboard panel, UI component, or API consumer may independently calculate state. Every interface must consume data directly from the Digital Twin API.

### 1.1 Core Principles
1.  **Immutability:** Once a state object is generated for a specific 5-minute bar, it is cryptographically immutable. It cannot be altered retroactively.
2.  **Canonical Truth:** If there is a discrepancy between a UI panel and the Digital Twin, the Digital Twin is correct by definition.
3.  **Complete Explainability:** Every object must contain the necessary metadata to explain its existence, its dependencies, and its validation status.

---

## 2. Object Model Definition

Every subsystem within Atlas is represented as a distinct object within the Digital Twin. All objects inherit from a base `AtlasEntity` class, which enforces mandatory metadata.

### 2.1 Base Entity Metadata
Every object must expose:
*   `entity_id`: Unique UUID.
*   `timestamp`: Unix epoch of generation.
*   `bar_index`: The TradingView bar index.
*   `owner_module`: The Pine module or backend service responsible for generation (e.g., `M-03`).
*   `aps_reference`: The specific section of the Atlas Production Specification governing this object.
*   `status`: Enum (`HEALTHY`, `DEGRADED`, `FAILED`).
*   `validation_status`: Boolean indicating if the object passed internal consistency checks.

### 2.2 Core Subsystem Objects

**1. MarketStateObject (MSO)**
*   **Purpose:** Represents the objective reality of the market.
*   **Properties:** Session, Regime, Trend Alignment, Volatility State, Active MVCs.
*   **Dependencies:** Raw OHLCV data.

**2. ModelEvaluationObject**
*   **Purpose:** Represents a specific model's interpretation of the Market State.
*   **Properties:** Model ID (e.g., A1), Eligibility (Boolean), Edge Score (Float), Rejection Rationale.
*   **Dependencies:** MarketStateObject.

**3. DecisionEngineObject (ADE)**
*   **Purpose:** Represents the candidate selection logic.
*   **Properties:** Ranked List of ModelEvaluationObjects, Selected Candidate, Selection Rationale.
*   **Dependencies:** All ModelEvaluationObjects.

**4. RiskIntelligenceObject (ARI)**
*   **Purpose:** Represents the capital allocation and safety rule evaluation.
*   **Properties:** Approved (Boolean), Risk Multiplier, Contract Size, Rule Evaluations (R1-R8 status).
*   **Dependencies:** DecisionEngineObject, PortfolioObject.

**5. VerificationObject (TVL)**
*   **Purpose:** Represents the final safety gatekeeper.
*   **Properties:** Verified (Boolean), Rejection Code, Rule Evaluation Matrix (38 rules).
*   **Dependencies:** RiskIntelligenceObject.

**6. InfrastructureObject**
*   **Purpose:** Represents the physical health of the system.
*   **Properties:** Webhook Latency, Broker API Latency, Database Connection Status, Pine Script Heartbeat.

## 3. Digital Twin API Specification

The Digital Twin API is a high-performance GraphQL or REST interface that serves the canonical state to all ANOS clients.

### 3.1 Endpoint Architecture

*   `GET /api/twin/state/current`
    *   **Returns:** The complete, unified state of all objects for the most recent 5-minute bar. This is the primary endpoint powering the Mission Control dashboard.
*   `GET /api/twin/state/historical/{bar_index}`
    *   **Returns:** The exact, immutable state of all objects as they existed at the specified historical bar. This powers the Time Travel Replay Engine.
*   `GET /api/twin/object/{entity_id}`
    *   **Returns:** Deep inspection data for a specific object, including its calculation inputs, outputs, and dependencies. This powers the Object Explorer.

### 3.2 Time Travel Capability
Because every object is immutable and stamped with a `bar_index`, the API natively supports Time Travel. When a user selects a historical candle in the UI, the frontend simply calls `/api/twin/state/historical/{bar_index}`. The API reconstructs the exact graph of objects that existed at that millisecond, allowing the UI to render the historical state with 100% fidelity.
