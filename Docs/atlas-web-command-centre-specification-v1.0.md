# Atlas Web Command Centre Specification v1.0

**Sprint:** 077  
**Date:** 10 July 2026  
**Author:** Manus AI  
**Status:** Approved for Implementation

The Atlas Web Command Centre is a read-only, browser-based observability dashboard. It provides real-time visibility into the Atlas Trading System (ATS) v2.1, allowing the operator to monitor the system's state, model evaluations, risk decisions, and execution status. The dashboard strictly enforces the principle of non-interference: it cannot place trades, modify Pine Script, alter the Atlas Production Specification (APS), or change live execution behaviour.

---

## 1. Architectural Overview

The Web Command Centre operates as an independent parallel workstream to the Pine Script execution path. It receives data via a dedicated webhook endpoint, entirely separate from the execution routing pipeline.

TradingView Pine Script transmits two simultaneous webhooks upon completing a bar evaluation:
1.  **Execution Webhook:** Routed to TradersPost for live market execution.
2.  **Observability Webhook:** Routed to the Atlas Web Command Centre backend for logging and dashboard updates.

The backend processes the incoming payload, persists it to a database, and broadcasts the state update to the React frontend via Server-Sent Events (SSE) or WebSockets.

### 1.1 Technology Stack
*   **Frontend:** React (Next.js) with Tailwind CSS for rapid, responsive UI development.
*   **Backend:** FastAPI (Python) or Node.js (Express) to handle high-throughput webhook ingestion and SSE.
*   **Database:** PostgreSQL (production) or SQLite (MVP) for immutable event storage.
*   **Realtime Protocol:** Server-Sent Events (SSE) for unidirectional backend-to-frontend state broadcasting.
*   **Hosting:** Local deployment for MVP, transitioning to a secure VPS (e.g., AWS EC2 or DigitalOcean) for production.

## 2. Graphical User Interface (GUI) Layout

The dashboard is designed as a dense, information-rich single-page application. It is divided into logical panels that mirror the internal architecture of the Atlas system.

### 2.1 Live Market State Panel
This panel visualises the 40-field Market State Object (MSO) computed by the Market State Engine (MSE) on every 5-minute candle.
*   **Temporal Data:** Current Session, Time (ET).
*   **Volatility Metrics:** ADX(14), ATR(14), Volatility State (Compressed/Expanding).
*   **Trend Metrics:** EMA Structure (9/21/50 alignment), Trend State.
*   **Overnight Context:** Overnight Direction, Overnight Range normalised by ATR(14).
*   **Participation:** Relative Transaction Activity proxy.
*   **MVC Status:** Active Minimum Viable Combinations (e.g., MVC-003).

### 2.2 Model State Panel
This panel displays the independent evaluation of all promoted execution models (A1, A3, B1).
*   **Model Status:** Eligible or Not Eligible based on current regime constraints.
*   **Edge Score:** The calculated score (0–100) for the current bar.
*   **Trade Parameters:** Proposed Entry, Stop, Target, and Expected R-Multiple.
*   **Compatibility:** Session and Regime match validation.
*   **Rejection Rationale:** Explicit reason if the model generated no signal.

### 2.3 Atlas Decision Engine (ADE) Panel
This panel visualises the ADE ranking protocol.
*   **Model Rankings:** Sorted list of all evaluated models.
*   **Candidate Selection:** The highest-scoring model exceeding the activation threshold (60).
*   **Tie-Break Logic:** Explanation of tie-resolution if multiple models score equally.
*   **NO_TRADE Rationale:** Explicit reason if no candidate is selected.

### 2.4 ARI Risk Panel
This panel displays the capital allocation and safety state managed by Atlas Risk Intelligence (ARI).
*   **Performance Tracking:** Daily P&L, Daily Trade Count, Consecutive Wins/Losses.
*   **Risk Parameters:** Base Risk, Active Risk Multiplier, Approved Contract Quantity.
*   **Safety Status:** Drawdown State, Circuit Breaker Status.
*   **Rejection Rationale:** Explicit ARI rule violation if the candidate trade is blocked.

### 2.5 Trade Verification Layer (TVL) Panel
This panel exposes the final 38-rule safety gatekeeper.
*   **TVL Status:** VERIFIED, REJECTED, DELAYED, or EMERGENCY BLOCK.
*   **Validation Failures:** Specific category failures and rejection codes.
*   **Signal Integrity:** Signal ID and Duplicate Prevention status.
*   **Payload Preview:** Raw JSON payload generated by Pine Script.

### 2.6 Execution Panel
This panel monitors the physical routing of the order.
*   **Routing Status:** Webhook transmission confirmation.
*   **Broker Acknowledgements:** TradersPost and Tradovate connection states.
*   **Order Details:** Fill Price, Slippage calculation, Active Stop, and Target levels.
*   **Position Status:** Current active position tracking.

### 2.7 Live Chart Panel
An embedded TradingView widget locked to the MNQ 5-minute timeframe. Where possible via the TradingView Lightweight Charts API, this panel will overlay:
*   Active Model annotations.
*   Signal markers for Entry, Stop, and Target levels.
*   Session boundary shading.
*   MVC active zone highlighting.

### 2.8 Decision Timeline
A chronological, scrolling feed of every 5-minute candle decision, providing an audit trail of the system's thought process.
*   Timestamp.
*   Market state summary.
*   Model scores (A1, A3, B1).
*   ADE, ARI, and TVL decisions.
*   Execution result.

### 2.9 Account Monitoring Panel (Future Integration)
A placeholder panel for future API integrations with brokerage and prop firm dashboards.
*   Tradovate, Apex, and TopStep account balances.
*   Daily P&L and remaining drawdown limits.
*   Open position reconciliation.

### 2.10 Research & Build History
A static reference panel displaying system metadata.
*   Current ATS, APS, and Pine Script versions.
*   Active and completed sprint tracking.
*   Promoted and rejected model inventory.
*   Engineering change log access.

### 2.11 Alerts & Health Panel
System infrastructure monitoring.
*   TradingView alert status.
*   Webhook receiver uptime.
*   TradersPost and Tradovate API connection status.
*   Last received Pine log timestamp and system heartbeat.

## 4. Database Schema

The backend database persists every received webhook to create an immutable audit trail.

**Table: `atlas_observability_logs`**
*   `id` (Primary Key, UUID)
*   `timestamp_utc` (Integer)
*   `bar_index` (Integer)
*   `market_state` (JSONB)
*   `model_evaluations` (JSONB)
*   `ade_decision` (JSONB)
*   `ari_decision` (JSONB)
*   `tvl_decision` (JSONB)
*   `execution_payload` (JSONB)
*   `created_at` (Timestamp)

## 5. Security Model

The Web Command Centre must not introduce attack vectors into the trading system.
1.  **Read-Only Architecture:** The dashboard contains no functionality to transmit orders or modify state.
2.  **Authentication:** The frontend requires strong password authentication. No public access is permitted.
3.  **Webhook Authentication:** The backend observability endpoint requires a static Bearer token matching the token configured in the TradingView alert.
4.  **Network Isolation:** The observability backend operates on a separate port or server from the execution webhook receiver to prevent resource exhaustion attacks affecting live trading.

## 6. Implementation Roadmap

### Phase 1: MVP Build (Local Deployment)
*   **Backend:** Stand up FastAPI server with SQLite database and a `/webhook/observe` endpoint.
*   **Frontend:** Scaffold React application with static layout for the 11 defined panels.
*   **Integration:** Implement Server-Sent Events (SSE) to push database inserts to the frontend in real-time.
*   **Pine Script:** Update M-12 (`atlas_observatory`) to generate and transmit the observability JSON payload.

### Phase 2: Chart Integration and UI Polish
*   Integrate TradingView Lightweight Charts API into the Live Chart Panel.
*   Implement colour-coded state indicators (e.g., green for VERIFIED, red for EMERGENCY BLOCK).
*   Build the scrolling Decision Timeline component.

### Phase 3: Production Deployment
*   Migrate database to PostgreSQL.
*   Deploy backend and frontend to a secure VPS.
*   Implement frontend authentication.

### Phase 4: Future Expansion
*   Integrate Tradovate and Prop Firm APIs for the Account Monitoring Panel.
*   Implement historical replay functionality to review specific trading days within the dashboard GUI.
